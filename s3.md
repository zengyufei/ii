# ii S3 中转方案

## 目标

给 `ii` 增加一个可选的 S3 中转模式，让用户在复杂网络、长链路或对端不方便直连时，先把文件上传到对象存储，再由 `ii recv <ticket>` 从对象存储拉取。

默认后端定为 Cloudflare R2。其他 S3 兼容存储通过 `ii.toml` 手动改配置支持，不要求用户每次运行都重新输入。

## 核心原则

- 命令行优先，不做网页表单。
- 首次配置时，允许在 `ii send` 的终端里逐项输入。
- 配置成功后写入 `ii.toml`，下次直接复用。
- `ii recv` 不接触 S3 密钥，只消费 ticket。
- ticket 里只放取回文件所需的信息，不放 secret。

## 用户流程

第一次运行：

```powershell
ii send .\video.mp4 --s3
```

如果还没有 S3 配置，`ii` 暂停当前发送流程，在同一个命令行窗口里提示用户完成初始化：

```text
ii: Cloudflare R2 is not configured.
Open this page:
https://dash.cloudflare.com/?to=/:account/r2/api-tokens

Account ID:
Bucket:
Access Key ID:
Secret Access Key:
```

输入完成后，`ii` 继续执行：

- 校验凭据
- 写入 `ii.toml`
- 上传文件到 Cloudflare R2
- 生成 ticket

第二次及以后运行：

```powershell
ii send .\video.mp4 --s3
```

如果配置已存在，就直接上传，不再询问。

## 配置文件

建议使用 `ii.toml` 保存 S3 配置。默认配置块只保留一个 Cloudflare profile：

```toml
[storage]
backend = "s3"
profile = "cloudflare"

[storage.s3.cloudflare]
provider = "cloudflare-r2"
account_id = "xxxx"
bucket = "ii"
endpoint = "https://xxxx.r2.cloudflarestorage.com"
region = "auto"
access_key_id = "xxxx"
secret_access_key = "xxxx"
prefix = "ii/"
presign_ttl_seconds = 86400
```

说明：

- `account_id` 用来拼 Cloudflare R2 endpoint。
- `bucket` 是文件落点。
- `prefix` 让对象名保持可管理。
- `presign_ttl_seconds` 控制 ticket 的有效期。
- `secret_access_key` 只写本地配置，不进入 ticket。

## ticket 设计

`ii send --s3` 成功后生成新的 ticket，ticket 只承担“让接收端找到对象”的职责。

推荐放入 ticket 的内容：

- 文件名
- 文件大小
- 内容 hash
- 对象路径
- 短时有效的预签名下载信息，或者等价的下载授权信息

不建议放入 ticket 的内容：

- `secret_access_key`
- 长期可用的对象存储管理凭据

## 交互细节

命令行交互要简单，顺序固定，用户只需要一路回车或填值。

建议顺序：

1. `Account ID`
2. `Bucket`
3. `Access Key ID`
4. `Secret Access Key`

补充规则：

- `Secret Access Key` 输入时不回显。
- 每输入一项，立刻进入下一项。
- 如果某项已经在配置里存在，就直接跳过。
- 如果校验失败，回到当前项重输，不要整段重来。

## 失败回退

如果自动初始化失败，`ii` 仍然要让用户能继续完成配置。

可接受的回退方式：

- 重新提示当前字段
- 显示缺失项
- 允许用户中断后再次执行 `ii send --s3` 继续补齐

不建议的方式：

- 直接退出并让用户手工猜配置字段
- 要求用户先编辑复杂 JSON
- 要求用户先做网页登录再回到命令行

## 其他 S3

Cloudflare R2 只作为默认 profile，不限制其它实现。

如果用户要换别的 S3 兼容后端，应该直接改 `ii.toml`：

- endpoint
- region
- bucket
- access key
- secret key
- prefix

## 设计取向

这个方案的目标不是把 S3 做成一个独立云存储工具，而是把它做成 `ii` 的一种传输后端。

所以体验上要遵守三条：

- 第一次麻烦一次，之后不再麻烦。
- 配置尽量少，字段尽量直白。
- 默认行为简单，手动改动留给高级用户。
